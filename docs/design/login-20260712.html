Login gate wireframe

2026-07-12 · Phase 1 session 1. One layout, four states. The gate is a state, not a place: it renders full-screen at whatever URL the user opened, and the OAuth callback reuses the same layout as its busy state.

Decisions

  • Gate, not prompt: the reference instance requires auth for timelines, so an unauthenticated shell would be all errors. Full-screen login at any URL; on success the same URL renders its real page (no redirect bookkeeping).
  • Identity is app name + location.host only. No /api/v1/instance fetch before login — it would add a failure mode to the very first screen for decoration. Phase 4 polish candidate.
  • The callback screen is not a separate design: state 3 is the same centered column. Errors from all three failure sources (registration, authorize denial, code exchange) land in the same inline message + retry button (errors travel as values, ADR-0008).
  • Centered single column, button in one-handed reach on mobile; semantic tokens only.
  • No scope explanation on this screen — the instance's own /oauth/authorize page shows what is being granted.

Open

  • None for this screen. Logout affordance lives in the app shell header (see app-shell wireframe backlog), not here.
